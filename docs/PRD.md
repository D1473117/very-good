# 產品需求文件 (PRD)

## 1. 專案概述
- **專案名稱**：隨機推薦系統 - 隨便吃什麼都好 (Random Meal Recommendation System)
- **背景與動機**：現代人每天面臨無數選擇，尤其在決定「下一餐吃什麼」時，常因選項過多而產生決策疲勞（Decision Fatigue）。為了解決這個日常痛點，我們希望透過開發一個隨機推薦平台，將使用者的「決策成本」降至最低。
- **目標用戶**：上班族與大學生（快速決定日常飲食）、群體決策者（解決聚餐不知道吃什麼的窘境）、喜愛探索者（嘗試沒吃過的新餐廳）、生活極簡主義者（不想花腦力在飲食決策上）。
- **核心價值主張**：一鍵解決飲食決策難題，節省時間與腦力，並帶來探索新美食的驚喜感。

## 2. 功能需求
以下為系統主要功能及其對應的使用者故事：

1. **一鍵隨機抽取推薦 (Random Draw)**
   - 作為 **猶豫不決的食客**，我希望 **能點擊一個按鈕就立刻得到一家餐廳或餐點的推薦**，以便 **我不需要自己做決定就能知道下一餐吃什麼**。

2. **條件篩選抽籤 (Filtered Random Draw)**
   - 作為 **有預算或距離限制的使用者**，我希望 **能在抽籤前設定距離範圍、價格區間與餐點類型（如：日式、素食等）**，以便 **系統推薦的餐廳符合我當下的實際需求**。

3. **餐廳詳細資訊與地圖導航 (Restaurant Info & Navigation)**
   - 作為 **準備前往用餐的使用者**，我希望 **能看到推薦餐廳的營業時間、評價、照片並一鍵開啟 Google Maps 導航**，以便 **我能順利抵達並了解餐廳基本狀況**。

4. **黑名單與我的最愛 (Blacklist & Favorites)**
   - 作為 **有特定偏好的使用者**，我希望 **能將踩雷的餐廳加入黑名單（不再推薦），並將喜歡的餐廳加入最愛**，以便 **系統的推薦能越來越符合我的胃口**。

5. **群組抽籤與結果分享 (Group Draw & Sharing)**
   - 作為 **群體決策者（如聚餐主揪）**，我希望 **能將抽籤結果快速分享給朋友，或邀請朋友一起進行排除與抽籤**，以便 **快速凝聚眾人共識，解決聚餐決定困難**。

6. **歷史紀錄 (History Tracking)**
   - 作為 **經常使用系統的用戶**，我希望 **能查看過去的抽籤與用餐紀錄**，以便 **回顧吃過的美食，或避免系統在短時間內推薦重複的餐廳**。

## 3. 非功能需求
- **技術限制**：後端採用 Python Flask 框架，前端使用 Jinja2 模板引擎與 HTML/CSS/JS 進行渲染，資料庫使用 SQLite 進行輕量級資料儲存。
- **效能考量**：隨機抽取的演算法必須在 1 秒內回傳結果，確保流暢的操作體驗。
- **安全考量**：使用者的最愛清單與黑名單需綁定個人帳號或 Session，確保個人隱私資料不外洩，並防止 SQL Injection 等常見網路攻擊。
- **UI/UX 設計**：介面需極簡且直覺（Mobile-first 設計），以符合「降低決策成本」的核心精神，抽籤過程可加入趣味動畫以增加驚喜感。

## 4. MVP 範圍 (Minimum Viable Product)

- **Must Have (必須具備)**
  - 核心的一鍵隨機抽取功能（可內建一份基礎的餐廳/餐點資料庫）。
  - 基礎的條件篩選（如：選擇吃飯、吃麵、或吃速食）。
  - 顯示餐廳基本資訊（名稱、類型）。

- **Should Have (應該具備)**
  - 簡單的使用者系統，用於儲存個人專屬的餐廳資料。
  - 黑名單與我的最愛功能。
  - 串接 Google Maps 連結以便導航。

- **Nice to Have (加分功能)**
  - 群組共同抽籤/投票功能與分享連結。
  - 酷炫的轉盤或抽籤動畫效果。
  - 飲食歷史紀錄與熱量/花費統計。
  - 串接第三方 API (如 Google Places) 自動獲取附近餐廳。
