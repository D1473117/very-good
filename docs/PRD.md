# 隨便吃什麼都好 系統 - 產品需求文件 (PRD)

## 1. 專案概述

### 背景與動機
現代人生活節奏快速，經常面臨「不知道要吃什麼」的選擇困難。這種日常的小決策往往消耗了大量的心力，甚至引發群體間的意見分歧。本系統旨在透過遊戲化的隨機推薦機制，消除這種「選擇疲勞」，將瑣碎的餐飲決策轉化為一場充滿驚喜的探索體驗，降低用戶的決策成本，帶來趣味性。

### 目標用戶
1. **上班族與大學生**：每天生活節奏固定，面臨選擇有限卻又容易感到厭倦的族群。
2. **群體決策者**：如部門小組長或聚會發起人，需要一個公正工具來快速達成用餐共識。
3. **喜愛探索者**：願意嘗試新口味，但缺乏搜尋動力或特定目標的消費者。
4. **生活極簡主義者**：希望減少日常瑣事干擾，將精力集中在更重要決策上的人。

### 核心價值主張
提供一個快速、公正、充滿驚喜的餐廳決策工具，讓「決定吃什麼」不再是煩惱，而是日常生活中的小樂趣。

## 2. 功能需求

以下為系統主要功能及其使用者故事（User Stories）：

1. **附近餐廳推薦（台中）**
   - 作為 **台中地區的上班族/大學生**，我希望 **系統能根據我的定位或指定區域推薦台中的餐廳**，以便 **我能快速找到附近可行的用餐選項**。

2. **隨機餐廳抽選**
   - 作為 **面臨選擇困難的使用者**，我希望 **能點擊一個按鈕進行隨機抽餐廳**，以便 **系統幫我做出客觀的決定，省去思考的麻煩**。

3. **條件篩選**
   - 作為 **有特定飲食偏好的使用者**，我希望 **能設定價格區間、餐點類型或距離等條件進行篩選**，以便 **隨機抽選的結果能符合我當下的需求和預算**。

4. **餐廳資訊顯示**
   - 作為 **決定前往用餐的消費者**，我希望 **能看到餐廳的詳細資訊（如地址、營業時間、評價等）**，以便 **我能確認該餐廳是否真的合適且正在營業**。

5. **收藏與歷史紀錄**
   - 作為 **經常使用系統的用戶**，我希望 **能收藏喜歡的餐廳並查看過去抽中的紀錄**，以便 **未來能直接回訪好吃的店或避免一直吃重複的餐點**。

6. **「多人投票決策」模式**
   - 作為 **聚會發起人**，我希望 **能發起一個多人投票活動並分享連結給朋友**，以便 **大家能在幾個隨機選項中快速投票，達成用餐共識，避免爭議**。

## 3. 非功能需求

### 技術選型
| 層面 | 技術選擇 | 說明 |
|---|---|---|
| 前端 | HTML / CSS / JavaScript（純前端，無框架） | 課堂統一規範 |
| 後端 | Python + Flask | 課堂統一規範 |
| 資料庫 | SQLite | 輕量級關聯式資料庫，無需額外安裝伺服器，非常適合小型專題與 Flask 搭配。 |
| 地圖/餐廳 API | Google Places API | 取得附近餐廳資訊、評分及經緯度的業界標準工具（有免費額度）。 |
| 前端樣式 | Bootstrap 5 | 快速達成 RWD 響應式設計，讓系統在手機與電腦上都能流暢使用。 |

### 效能、安全與體驗考量
- **效能**：系統需在 3 秒內完成附近餐廳搜尋與推薦結果顯示。
- **安全性**：使用者定位資料需經授權後使用，並避免長期儲存精確位置資訊；防範基本的 Web 攻擊（如 SQL Injection、XSS），且多人投票模式需具備防灌票機制。
- **易用性**：提供「一鍵推薦」功能，讓使用者能在 10 秒內完成選擇。
- **相容性**：系統需支援手機與桌面瀏覽器（RWD 設計）。

## 4. MVP (最小可行性產品) 範圍

### Must Have（必須擁有：核心功能）
- 基礎餐廳資料庫（以台中地區為主）。
- 單人隨機餐廳抽選功能。
- 基礎條件篩選（餐點類型、價格區間）。
- 餐廳基本資訊顯示（名稱、地址、營業時間）。

### Should Have（應該擁有：提升體驗）
- 「多人投票決策」模式（可建立專屬投票房間與連結）。
- 收藏喜愛餐廳與歷史紀錄功能。
- 抽選過程的遊戲化 UI（如轉盤或抽籤特效）。

### Nice to Have（最好擁有：未來擴充）
- 整合外部地圖 API（如 Google Maps）進行實時導航或準確距離計算。
- 用戶評論與心得分享系統。
- 特殊節日或主題式的美食探索包。

### 範圍限制（Out of Scope：本次不包含）
- **即時導航功能**：僅提供地圖連結，不會在系統內串接即時導航。
- **交易相關功能**：本次不包含外送、訂位或付款功能。
- **AI 深度學習推薦**：無個人化推薦模型，僅使用基本條件與隨機邏輯。
