# 產品需求文件（PRD）

**專案名稱：** 隨便吃什麼都好（Let's Just Eat）
**版本：** v1.0
**撰寫日期：** 2026-05-14
**狀態：** 草稿

---

## 1. 專案概述

### 1.1 背景與動機

在現代快節奏的生活中，「今天要吃什麼？」這個看似簡單的問題，卻每天消耗大量人們的心理能量。上班族與學生在午休有限的時間內需要快速做出決定，群體用餐時更容易陷入漫長的討論僵局。

本系統希望透過隨機推薦平台，將「決策成本」降至最低。透過數位化的隨機篩選機制，不僅能為使用者提供趣味性的互動體驗，也能打破人們總是吃固定幾間店的「味覺慣性」，探索更多潛在的餐食選擇，將瑣碎的煩惱轉化為輕鬆的遊戲化體驗。

### 1.2 目標用戶

| 用戶類型 | 描述 |
|---|---|
| **上班族與大學生** | 每天生活節奏固定，面臨選擇有限卻又容易感到厭倦的族群 |
| **群體決策者** | 如部門小組長或聚會發起人，需要一個公正工具來快速達成用餐共識 |
| **喜愛探索者** | 願意嘗試新口味，但缺乏搜尋動力或特定目標的消費者 |
| **生活極簡主義者** | 希望減少日常瑣事干擾，將精力集中在更重要決策上的人 |

### 1.3 核心價值主張

> 🎲 「不是不知道要吃什麼，而是懶得想。讓系統幫你決定。」

- **零決策成本**：一鍵隨機，秒速出結果
- **打破慣性**：發掘你從未嘗試的在地好店
- **群體共識**：多人投票，公平公正達成共識
- **個人化體驗**：收藏紀錄，越用越懂你

## 2. 功能需求

### 功能一：附近餐廳推薦

依據使用者目前的 GPS 位置，顯示附近一定範圍內（預設 500 公尺、可調整至 1km / 3km）的餐廳清單，並以距離排序顯示。

**使用者故事：**
- 作為 **上班族**，我希望 **系統能自動抓取我目前位置周邊的餐廳**，以便 **我不需要手動輸入地址就能快速看到附近選項**。
- 作為 **喜愛探索者**，我希望 **系統顯示我平常不常去的新餐廳**，以便 **我能輕鬆發現陌生但有趣的用餐地點**。

---

### 功能二：隨機餐廳抽選（台中）

提供「台中」為主要地區範疇的餐廳資料庫，使用者點擊「抽！」按鈕後，系統從符合條件的餐廳中隨機選出一間，並以動畫方式呈現結果，增加趣味感。

**使用者故事：**
- 作為 **大學生**，我希望 **按下一個按鈕就能得到隨機餐廳建議**，以便 **我在猶豫不決時能快速做出決定**。
- 作為 **生活極簡主義者**，我希望 **整個選擇過程在 5 秒內完成**，以便 **我不用花時間在瑣事上**。

---

### 功能三：條件篩選

使用者可在抽選前設定篩選條件，系統僅從符合條件的餐廳中進行隨機抽選。

**篩選維度：**
- 餐點類別（中式、日式、韓式、美式、素食、飲料…等）
- 價格範圍（$50 以下 / $50–$150 / $150–$300 / $300+）
- 距離範圍（500m / 1km / 3km / 不限）
- 用餐方式（內用 / 外帶 / 外送）
- 營業狀態（目前營業中 / 全部）

**使用者故事：**
- 作為 **群體決策者**，我希望 **能設定預算上限後再抽選**，以便 **確保推薦結果符合整個團隊的消費水準**。
- 作為 **素食者**，我希望 **能篩選只顯示素食友善的餐廳**，以便 **隨機結果不會出現無法接受的選項**。

---

### 功能四：餐廳資訊顯示

抽選出餐廳後，展示完整的餐廳詳細資訊頁面，讓使用者能立即獲得所有必要資訊。

**顯示內容：**
- 餐廳名稱、地址、電話
- 營業時間
- 餐點類別與平均消費
- 評分（1–5 星，來自使用者評分）
- 餐廳照片（封面圖）
- Google Maps 連結
- 最近的抽選人數（社群熱度）

**使用者故事：**
- 作為 **任何使用者**，我希望 **在抽選結果頁面看到餐廳的完整資訊**，以便 **我能立刻決定是否前往，不需要再另外 Google**。

---

### 功能五：收藏與歷史紀錄

登入後的使用者可以收藏喜愛的餐廳，並查看過去的抽選歷史記錄，幫助做出更好的決策。

**子功能：**
- 愛心收藏餐廳（加入我的最愛）
- 查看「我的最愛」清單
- 查看「抽選歷史」（時間戳記、當時抽到的餐廳）
- 從歷史紀錄中快速重訪餐廳頁面
- 排除已去過的餐廳（可選設定）

**使用者故事：**
- 作為 **喜愛探索者**，我希望 **系統記得我收藏的餐廳**，以便 **我能快速找到之前喜歡的地方**。
- 作為 **上班族**，我希望 **查看抽選歷史並標記「已去過」**，以便 **下次可以自動排除，確保每次都有新選擇**。

---

### 功能六：多人投票決策模式

發起人建立一個投票房間，邀請其他人加入，每人從系統推薦的餐廳候選清單中進行投票，最終由票數最高者勝出。

**流程：**
1. 發起人設定條件（地區、類別、價格範圍）並建立房間
2. 系統產生邀請連結（或 6 碼房間代碼）
3. 其他人以連結或代碼加入（無需登入）
4. 系統隨機提供 3–5 間候選餐廳
5. 每位成員投票（可投 1 票）
6. 倒數計時結束後，公布得票最高者
7. 同票時系統自動抽選

**使用者故事：**
- 作為 **群體決策者**，我希望 **能建立投票房間並分享連結給同事**，以便 **用餐決策過程既公平又有趣，不再是我一個人煩惱**。
- 作為 **聚會參與者**，我希望 **不需要註冊帳號就能加入投票**，以便 **低門檻地參與決策，不浪費時間**。

## 3. 非功能需求

### 3.1 技術限制

| 技術項目 | 規格 |
|---|---|
| **後端框架** | Python Flask |
| **模板引擎** | Jinja2 |
| **資料庫** | SQLite（開發環境）|
| **前端** | HTML5 + CSS3 + 原生 JavaScript |
| **地圖服務** | Google Maps Embed API / Geolocation API |
| **部署環境** | 本地開發為主（可選 Render / Railway 部署）|

### 3.2 效能需求

- 隨機抽選回應時間：**< 1 秒**
- 頁面首次載入時間：**< 3 秒**
- 多人投票房間支援：**最多 20 人同時連線**
- 資料庫餐廳資料量：**初始收錄台中地區至少 100 間**

### 3.3 安全考量

- 使用者密碼需經過 **bcrypt** 雜湊儲存
- 防止 SQL Injection（使用 SQLAlchemy ORM 或參數化查詢）
- 投票房間設有時效限制（最長 24 小時自動關閉）
- 敏感路由需要登入驗證（收藏、歷史紀錄）

### 3.4 使用性需求

- 介面需支援**行動裝置（Mobile First）**
- 主要操作流程（從開啟到拿到結果）**不超過 3 次點擊**
- 支援繁體中文介面
